## AC4 re-alignment (post-build-start)

### Why AC4 had to change

Aligned AC4 demanded "zero console errors ... including when relays/WebRTC are unreachable, as in the eval sandbox." Reproduced 2026-06-11 ~23:50 UTC (headless chromium, `--use-gl=angle --enable-unsafe-swiftshader`, default load of `http://localhost:3000/`):

- This sandbox's egress proxy intercepts TLS; every relay WebSocket attempt fails with `net::ERR_CERT_AUTHORITY_INVALID`. Chrome emits each failure as a console message of **type `error`** at the network layer ("WebSocket connection to 'wss://...' failed: ..."). No script writes it; try/catch and console patching cannot suppress it. Any implementation that attempts a network connection here produces these messages. The criterion as written fails every possible correct implementation, which makes it a broken criterion, not a high bar.
- Observed in the reproduction: 23 such messages, all matching `/WebSocket connection to .+ failed/`; **zero** other console errors; **zero** `pageerror` events (no uncaught exceptions, no unhandled promise rejections) over a 50 s watch. `data-net` settled to `"off"` by ~13 s; one info message ("presence relays unreachable, the museum is solo this visit"); last network-failure log at t=33.5 s with silence for the remaining ~16.5 s observed.

Two calibration facts baked into the amended bounds:

1. Network-layer WS failure messages DO carry a `location` (Chrome attributes the `WebSocket` constructor call site), so the verdict classifier is the **text pattern**, not "has no script stack".
2. The retry taper in the reproduction ran ~21 s past settle (backoff gaps growing: 15.3, 17.2, 21.1, 29.0, 33.5 s), not the ~15 s the builder reported. The cessation bound is therefore anchored at 45 s after page load and verified over a 60 s watch; a tighter bound would fail an honest backoff taper, a looser one would excuse real spam.

### Amended AC4 (replaces the aligned AC4 verbatim)

> **4. The production default settles honestly, degrades quietly, and the connection noise is bounded.** Loading the room with no `?net` param: within ~15 s of `data-loaded="true"`, `data-net` reads exactly `"webrtc"` or `"off"`, and from that deadline onward it never reads anything else and never changes again for the rest of the observation (no absent, blank, intermediate, or flapping values). When zero relay/WebRTC connections succeed — as in this sandbox, where all egress fails — it must settle to `"off"`; a `"webrtc"` claim with no live connection is a lying attribute and fails. Over an observation window extending to at least 60 s after page load: zero script-caused console errors and zero uncaught exceptions or unhandled promise rejections (Playwright `pageerror` events) at any point, and at most one info-level console message about presence. Browser network-layer connection-failure logs — console messages of type `error` whose text matches `/WebSocket connection to .+ failed/` (or that otherwise report only a failed connection attempt with a `net::` code, emitted by the browser network stack, not thrown or logged by script) — are exempt during settling but must be bounded and must cease: fewer than 50 such messages in total, none of them appearing later than 45 s after page load, so the final ~15 s of the watch are completely silent. In either settled outcome the solo room behaves identically to sprint 12: `data-peers="0"`, walking, focus, Enter to open a story, and Escape back all work.

### Verdict verification procedure (what I will actually run)

- Default load, console + `pageerror` listeners attached before navigation, watch >= 60 s from load.
- Settle check: `data-net` in {`webrtc`, `off`} by `data-loaded` + ~15 s; in this sandbox specifically, value must be `off`; re-sample through end of watch, value must never change after the deadline.
- Classify every console message: type `error` matching `/WebSocket connection to .+ failed/` or carrying only a `net::` connection code → exempt class; any other type `error` → instant fail; every `pageerror` → instant fail.
- Exempt-class bounds: total < 50; max timestamp <= 45 s after load; zero in the 45-60 s window.
- Info bound: presence-related messages below error severity <= 1 (framework dev noise — HMR, React DevTools, GL driver warnings — out of scope, as in prior sprints).
- Sprint-12 solo journey exercised on the settled page: walk, focus, `#museum-prompt`, Enter, Escape.

### Changes vs the aligned AC4 (rationale, one line each)

- **Exempted browser network-layer connection-failure logs (bounded):** physically unsuppressible under the sandbox TLS-intercepting proxy; demanding zero fails every correct implementation, so the class is exempted by an observable text-pattern classifier instead of dropped.
- **Added total (< 50) and cessation (none after 45 s, final 15 s silent) bounds:** "exempt" must not license infinite retry spam; bounds sit ~2x above the observed honest taper (23 messages, last at 33.5 s) and are directly measurable in the verdict.
- **Tightened: `data-net` must read `"off"` when zero relay connections succeed:** the original wording allowed a vacuous `"webrtc"` with nothing connected; an attribute sprint 14 builds on must not lie.
- **Tightened: no flapping after the settle deadline:** observed transient `"webrtc"` (4.8 s) → `"off"` (12.9 s) is fine inside the window, but post-deadline changes would make the contract unreliable for consumers.
- **Strengthened the script-error clause:** zero script errors and zero `pageerror` events now apply from load through the full 60 s watch (was: a 30 s window after settle), and the watch is longer — the exemption is paid for by a stricter everything-else gate.
- **Kept verbatim:** the ~15 s settle deadline, the at-most-one-info-message bound, and the sprint-12 solo behavior sentence.

ACs 1-3, 5, 6: untouched.
